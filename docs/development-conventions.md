# 开发与协作约定

> 文档定位：本文档用于统一代码修改、模块开发、板级适配和协作提交时的基本约定。

## 1. 基本原则

- 先找入口，再改实现。
- 先确认平台差异，再加条件分支。
- 先看模块边界，再决定放在哪个文件里。
- 先保证兼容，再考虑优化。

## 2. 代码组织约定

### 2.1 入口层

- `src/main.cpp` 只负责启动编排和主循环调度。
- 不要把复杂业务直接堆到 `setup()` 或 `loop()`。

### 2.2 模块层

- `src/modules/` 里每个模块尽量只做一件事。
- 新功能优先考虑是否能复用已有模块。
- 新模块需要在 `src/modules/Modules.cpp` 里明确注册。

### 2.3 平台层

- 板级差异优先放进 `variants/`。
- 只有确实跨板通用的逻辑，才放到公共层。
- 不要为了一个板子，把公共逻辑写成“到处都是 ifdef”。

## 3. 命名约定

- 类名使用清晰的职责名，例如 `RoutingModule`、`PowerFSMThread`。
- 文件名和类名尽量一致。
- 线程、状态、模块的名字应能直接反映职责。
- 常量命名要能看出它属于哪一层或哪一类资源。

## 4. 条件编译约定

- 能用运行时配置解决的，不要先写死成宏。
- 能用宏裁掉的平台能力，优先用宏控制。
- 新增 `#if` 时要确认对应的 `#else` 或默认路径。
- 条件越多，越要确认 `ESP32`、`nRF52`、`RP2040`、`Portduino` 的兼容性。

## 5. 状态机约定

- 状态机负责“状态”，不要塞太多杂活。
- 每个状态应有明确的进入、空闲和退出行为。
- 任何会改变供电、屏幕、蓝牙的操作，都要写清楚副作用。
- 状态切换应该能通过日志定位。

## 6. 模块开发约定

### 6.1 新增模块

新增模块时建议按这个顺序：

1. 定义职责。
2. 找到依赖对象。
3. 确认是否需要配置开关。
4. 决定是否由 `Modules.cpp` 创建。
5. 补充必要日志。

### 6.2 依赖管理

- 不要在构造函数里做无法回滚的大量 I/O。
- 对全局对象的访问要谨慎，避免初始化顺序问题。
- 模块之间通过已定义接口协作，不要互相偷看内部状态。

### 6.3 资源使用

- 主循环里尽量不要阻塞。
- 大量计算、轮询和外设操作应交给专门线程或任务。
- 共享资源要注意锁、信号量或访问时序。

## 7. 日志约定

- 启动流程要能看出跑到了哪一步。
- 失败日志要包含具体资源名、板型或地址。
- 高频循环日志要克制，避免淹没真正的问题。
- 状态切换建议使用统一前缀，便于检索。

## 8. 测试与验证约定

- 改启动逻辑后，至少确认能否正常进入主循环。
- 改模块装配后，确认没有破坏路由和消息处理。
- 改状态机后，确认睡眠、唤醒和供电切换都正常。
- 改板级逻辑后，确认对应变体的引脚和外设顺序没有变坏。

## 9. 提交前自检

- 是否影响其他平台编译
- 是否影响模块创建顺序
- 是否影响状态机副作用
- 是否需要更新文档
- 是否会让日志更难排障
