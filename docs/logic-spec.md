# Meshtastic Firmware 逻辑规范

> 文档定位：本文档用于说明工程的核心运行逻辑、模块边界、状态流和扩展约束，便于后续开发、联调和维护保持一致口径。

## 1. 工程定位

本工程是 Meshtastic 固件的本地源码仓库，核心职责是为不同硬件平台提供统一的 LoRa Mesh 通信能力，并在此基础上叠加以下能力：

- 设备启动和电源管理
- 无线通信与路由
- 屏幕和输入交互
- GPS、传感器和遥测
- 蓝牙、Wi-Fi、以太网和串口等外设能力
- 多硬件平台和变体适配

工程不是单一业务应用，而是一个“硬件能力组合平台”。因此，逻辑规范的重点不是页面流程，而是：

- 启动顺序是否正确
- 模块是否在正确时机创建
- 状态切换是否可预测
- 资源占用是否可控
- 平台差异是否被隔离

## 2. 目录职责

### 2.1 `src/main.cpp`

主入口，负责：

- 早期硬件初始化
- 电源安全检查
- 变体初始化
- 外设引脚和板级资源准备
- 屏幕、I2C、GPS、蓝牙等能力探测
- 创建全局状态对象
- 调用模块装配
- 启动电源状态机和后台线程
- 提供主循环 `loop()`

### 2.2 `src/modules/`

模块层，负责把功能拆成独立单元，例如：

- `RoutingModule`
- `TextMessageModule`
- `NodeInfoModule`
- `PositionModule`
- `WaypointModule`
- `Telemetry/*`
- `ExternalNotificationModule`

模块层的原则是“按能力拆分，不按 UI 拆分”。每个模块应尽量只负责自己的输入、处理和输出。

### 2.3 `src/PowerFSM.*`

电源状态机，负责设备在不同供电和睡眠场景下的状态切换，包括：

- `SDS`：深度睡眠
- `LS`：轻睡眠
- `NB`：普通工作状态
- `POWER`：外部供电状态
- `SERIAL`：串口工作状态
- `DARK`：屏幕关闭但蓝牙可用的状态

### 2.4 `variants/`

硬件变体目录，每个子目录代表一类板子或设备配置，通常包含：

- `platformio.ini`
- `variant.cpp`
- `variant.h`
- 板级引脚定义
- 必要时的自定义初始化逻辑

### 2.5 `boards/`

板级定义与构建配置数据，供 PlatformIO 和各个变体使用。

## 3. 启动逻辑规范

### 3.1 启动顺序

主启动流程应遵循以下顺序：

1. 初始化电源 HAL
2. 处理电源安全门槛，禁止低电量下误启动
3. 执行变体早期初始化
4. 配置板级电源、背光、按键和关键 CS 引脚
5. 执行屏幕/I2C/GPS/蓝牙等硬件探测
6. 创建全局状态对象
7. 装配模块
8. 启动电源状态机
9. 进入主循环

### 3.2 启动阶段约束

- 早期初始化必须先于任何依赖板级电源的外设访问
- `setupModules()` 必须在系统配置准备完成后调用
- 会影响电源状态的对象必须在状态机启动前可用
- 对屏幕、蓝牙、路由器等全局对象的访问应避免在“尚未初始化”阶段发生

### 3.3 平台分支

工程存在大量条件编译分支，因此任何启动逻辑修改都必须确认：

- ESP32、nRF52、RP2040、Linux/Portduino 等平台是否都能编译
- 不同板子是否需要不同的电源/外设顺序
- 某些功能是否依赖特定宏，例如 `HAS_SCREEN`、`HAS_WIFI`、`ARCH_ESP32`

## 4. 模块装配规范

### 4.1 创建原则

`src/modules/Modules.cpp` 负责模块实例创建。规范如下：

- 新模块要么在这里创建，要么由其他明确入口创建
- 模块创建时机应尽量晚于配置加载和硬件探测
- 是否启用某模块，应通过配置和编译宏共同控制

### 4.2 模块装配顺序

模块装配顺序不是随意的，当前工程中有几个重要约束：

- `StatusLEDModule` 需要尽早可用，便于系统状态反馈
- `RoutingModule` 必须最后创建，因为它会协调其他模块的回复和 ACK 行为
- 某些模块只在屏幕模式非彩色或特定角色下创建
- 遥测类模块通常依赖传感器扫描结果或模块配置

### 4.3 模块职责边界

模块应满足以下边界：

- 不直接修改全局硬件初始化顺序
- 不擅自接管其他模块的职责
- 不在构造函数里做过重 I/O，除非该模块设计明确要求
- 尽量把运行逻辑放到事件处理、轮询或线程中

## 5. 状态与数据流

### 5.1 全局状态对象

工程中有若干全局状态对象，用于跨模块共享：

- `PowerStatus`
- `GPSStatus`
- `NodeStatus`
- `BluetoothStatus`

规范要求：

- 全局状态对象只承载“状态”，不要承载复杂业务流程
- 状态变更应尽量通过明确接口通知相关模块
- 不要把临时计算结果长期塞进全局对象

### 5.2 典型数据流

#### 无线消息流

1. 接收无线消息
2. 路由层判断是否转发、应答或交给目标模块
3. 业务模块解析消息并更新状态
4. 必要时触发屏幕、串口、通知或遥测输出

#### 传感器遥测流

1. 启动时扫描 I2C/传感器
2. 根据探测结果和配置创建遥测模块
3. 模块定期读取数据
4. 数据经过单位转换、筛选和封装后上报

#### 电源状态流

1. 电源 HAL 读取外部供电和电池状态
2. 状态机判断当前应处于哪类电源状态
3. 进入睡眠、唤醒、蓝牙开关、屏幕开关等动作
4. 主循环持续检查电源命令和恢复逻辑

## 6. 电源状态机规范

### 6.1 状态定义

- `SDS`：深度睡眠，目标是最低功耗
- `LS`：轻睡眠，允许周期性唤醒
- `NB`：正常运行，适合无外部稳定供电场景
- `POWER`：外部供电运行，适合插电或持续供电设备
- `SERIAL`：串口交互优先
- `DARK`：屏幕关闭，但系统仍可交互

### 6.2 状态切换规则

- 供电状态变化应及时驱动状态机切换
- 外部供电可用时，设备应偏向 `POWER`
- 电池不足时，应优先进入更保守的睡眠策略
- 串口、按钮、IRQ、定时器等唤醒源都必须明确归类

### 6.3 状态动作要求

- 进入睡眠前必须尽量完成必要的预处理
- 退出状态时不要遗留蓝牙、屏幕或外设在错误状态
- 状态切换中的副作用必须可追踪，最好能通过日志定位

## 7. 主循环规范

`loop()` 是整个系统的持续执行入口，要求：

- 保持短小、可预期
- 只做周期性检查和调度，不做长时间阻塞任务
- 将耗时工作交给专门线程、状态机或后台模块

主循环当前应关注：

- 平台专用循环处理
- 电源命令检查
- 跳变 IRQ / 遗漏中断检查
- 服务层轮询
- 输入事件队列处理
- 端口/网络恢复逻辑
- 显示刷新和消息持久化相关任务

## 8. 并发与线程规范

工程使用线程/任务模型来分担主循环压力。规范如下：

- 线程名应明确反映职责
- 线程之间通过队列、状态对象或同步原语通信
- 尽量避免在主线程里直接做重工作
- 对共享资源的访问必须考虑互斥或时序

常见并发关注点：

- 串口控制台
- 电源线程
- 屏幕刷新线程
- 传感器轮询线程
- 网络或路由相关后台线程

## 9. 配置与条件编译规范

### 9.1 配置来源

逻辑配置主要来自：

- PlatformIO 构建环境
- `configuration.h` 和相关生成配置
- 运行时配置对象
- 板级 `variant` 文件

### 9.2 编写规则

- 不要把纯运行时判断写成硬编码板型分支
- 能通过配置控制的行为，优先走配置
- 能通过条件编译裁剪的能力，优先走宏开关
- 新增宏时要考虑默认值和兼容性

### 9.3 兼容性要求

任何新逻辑必须考虑：

- 无屏幕设备
- 无蓝牙设备
- 无 I2C 设备
- 无 GPS 设备
- 仅串口设备
- 仅网络设备

## 10. 扩展开发规范

### 10.1 新增模块时

新增模块建议遵循：

- 先定义清晰职责
- 再实现头文件和实现文件
- 再决定是否需要全局实例
- 最后在 `setupModules()` 中注册

### 10.2 新增硬件变体时

新增变体建议遵循：

- 复制最接近目标硬件的模板
- 单独确认引脚、电源和屏幕接线
- 检查是否需要额外的早期初始化
- 补齐对应 `platformio.ini`

### 10.3 新增状态或事件时

- 事件名应语义明确
- 状态切换要有进入、退出和空闲行为说明
- 每个事件必须有清晰的归属模块
- 不要把多个无关行为混进同一个事件

## 11. 错误处理规范

- 尽早失败，避免静默进入错误状态
- 对外设探测失败要有明确日志
- 对不可恢复错误，应进入可见的降级或重启路径
- 恢复逻辑不能反复占用主循环

## 12. 日志规范

日志应服务于三件事：

- 排查启动问题
- 排查状态切换问题
- 排查消息/传感器/网络异常

建议：

- 状态切换使用固定前缀
- 硬件探测失败打印具体设备和地址
- 模块装配问题打印启用条件
- 避免无意义高频日志污染串口

## 13. 变更检查清单

在修改核心逻辑前，建议至少确认：

- 是否影响启动顺序
- 是否影响 `setupModules()` 的模块创建顺序
- 是否影响电源状态机
- 是否影响某个平台的条件编译
- 是否需要新增或调整日志
- 是否需要补充板级变体配置
